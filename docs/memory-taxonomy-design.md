**Update, same day, post-v1.1.0:** the first real-world migration at meaningful scale (a different, larger project, 380 files/1.55MB) surfaced two more gaps that this project's own small memory dir couldn't have exposed: the single-shot classify/write design didn't scale past a real single-response budget, and the topic-scaffold recognition check didn't accept the metadata-nested frontmatter shape Claude Code's own tooling actually produces. Both fixed — see "Scaling past one subagent's output budget" below and the Implementation notes. This follow-up patch went through the same 3-reviewer pre-deploy ritual as v1.1.0 (code-review + pentest, Sonnet; Fable standing in for gemini). All three independently converged on the same critical defect from different angles — the batched path's rollback logic inferred which files it had created by scanning the directory for reserved topic filenames, "by construction" assumed none could have pre-existed, and that assumption was false whenever config and disk state could desync (a scenario this codebase already handles elsewhere, in `memory-apply.md` and `restore.md`, but this new path hadn't). Fixed with an explicit pre-flight collision check plus switching rollback tracking from disk inference to an explicitly-passed list of what this run actually wrote — see Implementation notes for the full account. Several smaller findings (no failure contract for write subagents, a byte-threshold gate calibrated on the wrong constraint, a portability gap, a one-directional conservation check) were fixed alongside it.

**Update, 2026-07-07, post-v1.4.0:** the LLM-write migration design described below (Phase B write subagents reproducing source content verbatim) has been **replaced**. Two real runs on a dense ~280KB directory rolled back (124 then 91 lines summarized away; the conservation check caught both, nothing lost) — proving that asking an LLM to byte-copy dense forensic prose is fundamentally unreliable, and no prompt wording fixes it. The new design splits the job: **the model places** each source file under one topic (judgment, JSON plan, no content reproduction); **a deterministic script moves the bytes** (`hooks/migrate-move.sh` — `cat` under a per-source heading). Conservation is now structural (cat can't summarize), not a hope about model behavior. The single-shot/batched split, write-chunking, the write prompt, and three of the four chunk thresholds are gone (`commands/clean.md` 1843→1461 lines). The migration is now **shape-agnostic** — topic files are verbatim source under per-source headings; shaping into the curated conventions (worklog datestamps, decisions headings) is **curate's** job afterward. It is also **resumable**: the JSON plan is saved to `.mempenny-migration-plan.json` and a re-run resumes from Phase B without re-paying classification. The hardened conservation-check awk script is unchanged. The trade is deliberate: lossless first, pretty later via curate.

**Update, 2026-07-03, post-v1.1.3:** a third real-world migration (a different, larger project, 75 files/728KB) surfaced a Phase B write-chunk scaling gap the v1.1.2 fix didn't fully close, and a conservation-check accuracy bug serious enough to roll back a migration that had actually succeeded. Both fixed, and both — plus everything the pre-deploy review round found on top of the first draft of these fixes, including two real security gaps in the new mechanical-append script — are recorded in full in the Implementation notes below, in the same honest, non-compressed style as the curate/shard-roll review history. Headline changes: Phase B write-chunking now has its own, tighter `{WRITE_CHUNK_CAP}` separate from Phase A's `{CHUNK_SIZE_CAP}` (see "Scaling past one subagent's output budget" below); the conservation check's frontmatter-stripping and exact-line-only comparison were both replaced with more robust equivalents; and the write-landing script gained symlink guards and stopped relying on shell state surviving between separate tool calls, among other fixes.

## Background

MemPenny's current auto-memory layout is one flat MEMORY.md index per project, pointing at individually-typed .md files (user/feedback/project/reference). This spec replaces it with a fixed, 3-level topic taxonomy, plus an automated migration path so every project converts to it without manual work.

## 1. The 8 topic files

| File | Purpose |
|---|---|
| charter.md | Goal + requirements for the artifact — small, stable, merged because always read together. |
| pending.md | In-flight work; volatile, overwritten freely; never sharded. |
| worklog.md | Datestamped log of completed/shipped changes to the artifact. |
| support.md | One log of helping people/systems operate the artifact; ages out by calendar, never "migrates" between files. |
| traps.md | Discovered hazards: condition + consequence + how to avoid. |
| rules.md | Standing directives for the workers — rituals, approvals, escalation. |
| decisions.md | ADR-style: why X over Y, alternatives rejected. |
| reference.md | Who/what-is-X: people, systems, URLs, invocation strings. |

Optional `howto.md` only if a procedure has no other home; it's the only permitted 9th MEMORY.md line, and migration never creates it.

**Disambiguators:**
- Requirement vs. rule: constrains the artifact vs. constrains the workers.
- Trap vs. rule: discovered hazard with a condition vs. directive without context. A trap promoted to a rule lives only in rules.md and may cite the trap.
- Worklog vs. support: changed the artifact vs. helped someone operate it.

**Canonical-home relaxation:** no requirement that an item have exactly one home. The same thing linked/mentioned from multiple topic files is expected, not a defect. Cross-references use `[[name]]`, targeting a heading's slug — link targets are `###` headings (traps, rules, reference entities, decisions) and topic filenames. Worklog/support list items are not link targets, by design.

## 2. Sharding rule

Only log-topics shard: worklog, support, decisions. Reference-topics (charter, rules, traps, reference, pending) never time-shard — over the ceiling they're reduced in place via **curate** (§4), or split by named sub-topic (e.g. `rules-prod.md`) as a deliberate human decision if genuinely irreducible.

- **Trigger:** 25KB or 200 lines, whichever first (matches Claude Code's own eager-load ceiling). Entry count (~50) is a review trigger, not a split trigger.
- **Mechanism:** a closed calendar year becomes `topic-YYYY.md`, flat, no subfolders, with the existing `mempenny-lock` marker on line 1 as the freeze mechanism (reuses shipped lock surface, zero new code for the freeze itself).
- **Index:** MEMORY.md only ever lists the 8 topic names — it never grows. Each sharded topic carries its own `## Shards` block (and `## Sub-files` block if a sub-topic split exists) at the top of the file, after any lock/title line, before the newest month. Sub-files (`rules-prod.md`, `howto.md`) are indexed there, never in MEMORY.md.
- **Pin:** shard-roll fires only at closed-year granularity. An open year that alone exceeds the ceiling is flagged in clean's report and tolerated — no mid-year shards, ever.
- Worst case for a cold agent: 3 deterministic file-opens to find anything.

## 3. Internal entry conventions

- **Log-topics** (worklog, support, decisions): entries grouped under lazily-created `## YYYY-MM` headings inside the current file (open `topic.md` or a closed `topic-YYYY.md` shard) — newest month first, newest entry first, never pre-create empty months.
  - worklog/support: `- **YYYY-MM-DD** — summary.` + optional 1-2 sentences. No other required fields.
  - decisions: `### YYYY-MM-DD — <greppable title>` (e.g. "nap non-interactive by design"), body beneath. Headings, not list items — this is what makes them linkable via `[[name]]` and lookup-by-topic rather than only by date.
- **traps.md / rules.md:** each entry is `### <short name>` containing the existing rule → **Why:** → **How to apply:** shape already used for today's individual feedback memories — reused, not reinvented.
- **reference.md:** `### <entity name>` + loose free-form lines (what it is, where/how to reach it). No rigid template — entity types vary too much for one shape to fit all.
- **charter.md / pending.md:** plain prose, no structure. Exempt from all automated reduction (no auto-curate, no auto-distill — distilling requirements is destructive). Over-ceiling is only flagged for human attention.

## 4. The "curate" operation (new capability)

Per-entry curation for over-ceiling reference-topics. Distinct from `memory-distill`, which is file-granularity and would be destructive on a multi-entry file (collapsing 15 independent rules into 3 sentences). Curate walks a file's individual `###` entries and applies memory-hygiene's existing keep/archive/delete judgment per entry — one heading = one decision.

- Runs through the same propose-table → apply-with-backup pipeline as triage/apply (unattended under nap's `--yes`, same as everything else — no special-casing).
- Entry-level ARCHIVE reuses the existing archive-destination convention at entry granularity.
- Skips locked files, per existing lock semantics.
- Triggered when clean finds a reference-topic over the ceiling.

## 5. Migration / auto-conversion design

**Config additions** to `~/.claude/mempenny.config.json` (v2 schema), per memory dir — see `commands/clean.md` Step 4 for the authoritative schema doc and validation rules:

- **`memory_layout`** — `"flat"` (default/absent — not yet migrated) or `"topics"` (migration complete). Explicit detection marker, never inferred from filenames present. Written only after a successful migration's conservation check passes; re-synced by `/mempenny:restore` to match whatever layout it actually restored.
- **`migrate_documents`** — boolean, default `true` when absent. Set `false` for a memory dir to opt it out of auto-migration permanently. Sits above the existing per-directory `.mempenny-lock`: the lock blocks migration for one run, this flag blocks it until a human flips it back.

**Scope, computed once, up front:** the orchestrating `/mempenny:clean` run lists every `.md` file directly under the memory dir (excluding `archive/`, **including `MEMORY.md` if present**) via a real directory listing before spawning anything — this list (`{OLD_FILES}`) is what the end-of-run conservation check verifies against. It is not reconstructed later from a classification subagent's self-reported output, because that indirection is exactly what let `MEMORY.md` go unaccounted for in the first implementation (see Implementation notes).

**Classification:** a read-only subagent (`subagent_type: Explore` — same structurally-can't-write pattern as triage) reads every file in the old flat layout and proposes a migration table: old file/content → target new topic file + section.

**Scaling past one subagent's output budget:** the classification-and-write design above works as a single pair of subagent calls only while the directory is small. Three distinct thresholds gate this, deliberately not the same number: `{SINGLE_SHOT_CEILING}` (35,000 bytes) decides whether the *whole directory* can go through one classify response and one write-and-verify response; `{CHUNK_SIZE_CAP}` (150,000 bytes) bounds how much source content one *classify* call reads within the batched path (its output is placement-only, so this is a read-side, not write-side, constraint); `{WRITE_CHUNK_CAP}` (65,000 bytes AND at most 9 source files, whichever binds first) bounds how much one Phase B *write* call relocates. Write chunking needed its own, tighter cap because a write subagent's real cost scales with how much it individually relocates and cross-verifies, not just raw bytes — a live large-directory migration hit real output-token-ceiling failures on `{CHUNK_SIZE_CAP}`-sized write chunks even with the append mechanism (below) working as designed; splitting the same failed chunk smaller, on both byte and file-count dimensions, consistently resolved it. See Implementation notes for the full account, and how the single-shot/batch-classify threshold split was found and fixed in an earlier round. Above `{SINGLE_SHOT_CEILING}`, `clean.md` Step 4b switches to a three-phase batched path: **(0)** first, a collision pre-check — since `memory_layout` is a machine-local config value that can desync from actual disk state, every file is checked against the topic-scaffold rubric before anything else; any reserved-name file already on disk aborts migration (or resyncs the config if the whole directory already looks migrated) rather than risking a rollback or conservation check that treats a pre-existing file as something this run created; **(1)** the file list is packed into `{CHUNK_SIZE_CAP}`-sized batches and classified in parallel by multiple Explore subagents that each propose *placement only* (no content reproduction, so batch count doesn't affect output size); **(2)** the merged placements are grouped by target topic and written by one `general-purpose` subagent per topic (further split into sequential, same-topic-only chunks bounded by `{WRITE_CHUNK_CAP}` if a single topic's content itself exceeds it) using a mechanical fragment-file-plus-Bash-append pattern rather than a subagent ever reading and reproducing the growing target file itself (see Implementation notes) — every write subagent returns an explicit `WRITE OK`/`WRITE FAILED`, and any failure aborts the whole migration and rolls back rather than proceeding on an incomplete picture; **(3)** one final subagent runs the same scripted bash conservation check as the small-directory path — checking both directions (missing old content, which fails the run after a word-coverage fallback that tolerates legitimate reordering; and new content that isn't traceable to anything old, reported but non-fatal) — across everything, then commits. The conservation check itself never needed batching — it's a shell text comparison, not a model response, so it scales to any directory size in one call regardless of how many subagents did the writing. Its rollback list is passed in explicitly from what Phase 2 actually wrote, never re-derived by scanning the directory afterward — the same reasoning as the collision pre-check in (0).

**`/mempenny:memory-curate` and `/mempenny:memory-shard-roll` — fixed differently from migration, not by porting migration's batching.** Both had the same theoretical output-budget ceiling migration hit, made *more* likely to trigger by migration's own fix (a live test produced a 525KB `rules.md`, which `clean.md`'s Steps 12b/12c would auto-trigger curate or shard-roll against on the next run). But unlike migration — where deciding *which topic* a piece of content belongs to is genuinely semantic and needs an LLM's judgment — shard-roll's split points (`## YYYY-MM` headings) and curate's entry boundaries (`### ` headings) are both exactly, mechanically, grep-able. Neither command actually needs an LLM to *move* content, only to *decide* what to do with it (already a separate, small-output step in both: shard-roll's Step 4 already just lists which years are closed; curate's Step 4 already just produces a verdict-only table, no content reproduced). So instead of migration's multi-subagent parallel-batch pattern, both apply steps were rewritten to run one Bash script that finds heading line numbers, computes ranges, and uses `sed -n 'START,ENDp'` to extract directly into the new files — the model never reads a large file into its own context and regenerates it through a Write call, which is the actual mechanism that bounded output size in the first place. This has no meaningful size ceiling at all (bash line-range extraction doesn't care if a file is 2KB or 20MB), verified byte-for-byte both directly and via real subagents executing the exact prompt text. See Implementation notes for the fail-safe structural guard this added (shard-roll) and the fail-toward-keep behavior for entries missing from the classification table (curate).

**Move-only vocabulary (binding):** the migration table's only verb is *relocate verbatim*. No delete, no rewrite, no summarizing. A source file may split into multiple entries at heading/bullet boundaries; each old file's content defaults to landing under a `###` heading named from its filename stem (preserves provenance, gives it a stable link target). Unplaceable content goes verbatim under `## Unsorted (migrated YYYY-MM-DD)` in reference.md rather than being dropped or guessed at. Dedup/merge/tidy is curate's job later, through the reviewed pipeline — migration never does it.

**Apply:** reuses the existing backup → verified file count → SHA256-manifest-before-any-destructive-action → restore machinery verbatim.

**No confirmation gate** — ratified by Fable, conditional on:
- **Move-only** (above) — worst case is misfiled content, never lost content.
- **Conservation check:** after writing the new layout and before removing anything old, mechanically verify every normalized non-empty line of every old file (including old MEMORY.md's own per-file summary bullets, which are real content) appears somewhere in the new layout. Fail → auto-restore, `memory_layout` marker left unset, loud visible error. Marker written last, only after verification passes (crash-safe: a half-run re-detects the old layout next time and the restore covers the rest).
- **Post-migration report, always:** N files → 8 topics, placement summary, explicit backup path, one-line restore command. No gate *before*; full transparency *after* — this is what preserves informed awareness without relying on an uninformed rubber-stamp.
- **Any lock aborts migration entirely** (folder-level `.mempenny-lock`/`.mempenny-fixture`, or a file-level `mempenny-lock` comment anywhere in the dir) — report why, don't attempt a partial/mixed-layout migration.

**Trigger scoping:**
- Only `clean` (including nap → clean) performs migration writes. `triage` stays read-only and reports "old layout — will migrate on next clean." `restore` never migrates.
- A migrating run is exclusive: migrate, report, stop. Triage/clean-proper happens on the *next* run — one structural change per unattended run.
- If the backup folder isn't configured yet for this dir, the existing first-run backup-folder prompt still runs first (configuration, not a consent gate on the migration itself).
- Empty or never-touched memory dirs skip classification entirely and are scaffolded directly: all 8 files created with a one-line purpose header each, so the fixed index is always truthful from the start.
- **Restore/marker sync:** backup manifests record `memory_layout` at backup time; restore sets the dir's marker to match what it restored, so restoring a pre-migration backup doesn't leave the marker permanently lying about the layout.

**Build-order constraint (blocking, not follow-on work):** the topic-aware triage/clean rubric — never DELETE or DISTILL an empty/near-empty scaffold topic file; cluster analysis stays within same-topic boundaries — must ship *before or with* auto-migration. Without it, the first post-migration nap-triggered clean sees 8 mostly-empty scaffolds and could delete them under today's rubric, destroying the very structure that was just built.

## Implementation notes

- Apply must fail-safe when a table references files that no longer exist (e.g. a pre-migration triage table applied post-migration): refuse, demand re-triage, never partially apply.
- No new concurrency guarantees — same single-session assumption as existing clean.
- `/mempenny:memory-curate` and `/mempenny:memory-shard-roll` are standalone commands, triggered automatically by `/mempenny:clean` Steps 12b/12c when a topic file crosses the ceiling, but also directly invocable by a user against any topic file.
- A pre-existing, unrelated bug was found and fixed during this implementation: the C1 path-validation regex (`^/[A-Za-z0-9/_.\- ]{1,4096}$`) and the L2 glob regex were malformed POSIX bracket expressions that never matched anything when executed literally via `grep -E`/bash `[[ =~ ]]` — most seriously in `hooks/nap-check.sh`, where it silently caused the SessionStart hook to exit before ever checking a nap schedule. Fixed everywhere by repositioning `-` to the end of each character class and moving `]` to the front where it needs to be literal. This means `/mempenny:nap` may never have actually fired for any user before this fix.
- **2026-07-02, found via real dogfooding on a much larger project ("loop", 380 files / 1.55MB), not caught by the v1.1.0 pre-deploy review because that review only had this project's own small memory dir to validate against:** the single-shot classify-and-write design (one subagent reproduces every file's content verbatim in one response) required ~390K tokens of output at that size — far past any viable single-turn budget. Fixed by the three-phase batched path described above in "Scaling past one subagent's output budget," gated on a 150,000-byte threshold measured from a real directory listing before any subagent is spawned. The workaround that surfaced this (10 placement-only classify batches, 26 size-capped write chunks, parallel across topics / sequential within a topic) was proven out live before being generalized into `clean.md` itself.
- **Same session:** the topic-scaffold recognition check (`clean.md`, `memory-triage.md`) originally required a *top-level* `type:` frontmatter field. Every real topic file this project has dogfooded already carries `type:` nested under a `metadata:` key instead — Claude Code's own native memory tooling rewrites plain frontmatter into that shape shortly after MemPenny writes it. Fixed by having the check accept `type:` in either position. MemPenny's own writes (scaffold, migration, curate, shard-roll) still emit the plain top-level form — only the *recognition* side needed to tolerate both.
- **curate/shard-roll's mechanical rewrite went through three rounds of review before shipping, each round finding real, non-contrived issues in the round before it — worth recording honestly rather than compressing to "and then it was fixed":**
  1. First pass (code-review + pentest + Fable) found and fixed: a heading-shaped line inside a fenced code example being mistaken for a real structural boundary (added fence-aware detection); shard-roll orphaning already-written locked shard files if a later year in the same batch failed (added cleanup on every failure path, not just the last one); `wc -l` and `while read` both silently dropping a file's last line when it lacks a trailing newline (added up-front newline normalization); curate tolerating a stale/mismatched classification-table row the same as a legitimate lock override (split into hard-fail-immediately vs. tolerated-to-5%); an archive-file symlink guard that only checked the parent directory, not the specific predictable file path; a `## Shards`/`## Sub-files` block silently inheriting whatever verdict a later entry got, if positioned after the first entry; shard-roll's monotonicity guard not separately checking the *topmost* year against the declared current year; unvalidated year tokens reaching path construction.
  2. A second, narrower round verifying those fixes actually closed found two more: **curate's per-heading text lookup silently conflates two entries that share byte-identical headings** — the second entry's table row overwrites the first's in the lookup map, so *both* physical entries get whichever verdict was listed last, and neither the invariant nor conservation check catches it (both are derived from the same corrupted range sets). Fixed by refusing to run at all if any two entries share a heading. Also found: the fence-awareness discipline from round 1 didn't propagate to two guards added *by* round 1 itself (the Shards/Sub-files positional check, and the invariant check's own re-scan) — both used plain `grep`, false-positive-failing a legitimately fenced quoted example. Fixed by making both fence-aware too.
  3. A third, adversarial round found the deepest issue: **the whole-file fence-balance check (even/odd `` ``` `` count) is necessary but not sufficient.** Two independent, ordinary authoring mistakes — an unclosed fence in one entry, and a separate stray/unclosed fence elsewhere — can cancel out to an even total while the fence-tracking state is wrong for everything in between, silently swallowing a real heading into the preceding entry's range. This is empirically real (reproduced in both scripts) and, on investigation, **not fully fixable by better pattern-matching** — a genuinely intentional quoted example heading and a real heading swallowed this way are observationally identical to any line-based tool, fence-aware or not. See "Known limitation" below for how this was handled instead of chased indefinitely. The same round also found: the final commit (`mv` the working copy over the real topic file) wasn't guaranteed same-filesystem, so a `mktemp` default location on a different device could fail the rename after the archive/shard write had already landed, with no clear FAILED signal (fixed: `mktemp` now targets the memory directory itself); and a pre-existing archive file lacking its own trailing newline would get the newly-appended content merged onto its last line (fixed: same normalization now applies to the archive file too, not just the topic file).
- **Known limitation, accepted rather than chased further:** curate and shard-roll's fence-aware structural detection cannot, even in principle, distinguish "a heading-shaped line legitimately quoted inside a fenced example" from "a real heading swallowed because two unrelated fence mistakes elsewhere in the file happened to cancel out" — both produce the exact same sequence of fence-toggle states to a line-based scanner. Fully closing this would require an actual markdown-aware parser, which is disproportionate to the risk (it requires two independent, non-adjacent authoring mistakes in the same file) given everything else in this diff already fails closed. Mitigated, not solved: both scripts now report `EXCLUDED_AS_FENCED` — a count of heading-shaped lines the fence-aware scan treated as inside a fence — in their success output, so an unusually high count is at least visible to whoever reads the report, even though the script itself can't tell which case it is.
- **Follow-up patch, pre-deploy review found before shipping:** the first draft of the batched path's finalize prompt determined which files were safe to delete-on-rollback (and which old files to remove on success) by scanning the directory for the 9 reserved topic filenames and asserting "by construction" that any which existed were created by the current run. All three reviewers (independently, from three different angles — a config/disk-desync scenario, a retry-after-interrupted-migration scenario, and a scenario tracing through this codebase's own existing disk-vs-config precedent in `memory-apply.md`/`restore.md`) found this assumption false: `memory_layout` is config, not disk state, and the two can desync. A pre-existing or leftover reserved-name file could self-satisfy the conservation check (its own lines match themselves) and then get deleted as part of either the success or failure path. Fixed two ways: **(a)** a collision pre-check in Step 4b.1 now inspects disk reality against the topic-scaffold rubric before any migration decision is made — a clean directory proceeds normally, an already-fully-migrated one resyncs the config without touching files, and a genuinely ambiguous mixed state aborts and asks a human to resolve it; **(b)** `{NEW_FILES}` — what's safe to roll back or treat as "this run's output" — is now built explicitly from confirmed `WRITE OK` returns during the run, never re-derived from a directory scan afterward, matching the pattern the single-shot path already used correctly. The write prompt also gained an explicit `WRITE FAILED` return path and CREATE/APPEND existence preconditions (it previously had neither, unlike every other subagent in this feature), and the conservation check in both paths now verifies in both directions — missing old content (fails the run) and unexplained new content (reported, not fatal) — where previously only the missing direction was checked. `find -printf`/`mapfile` (GNU/bash-4-only) were also replaced with a portable equivalent, since a BSD `stat` fallback two lines away implied portability was already meant to matter here.
- **Caught by testing the fix, not by review:** the collision pre-check's first draft defined the "already fully migrated, config just stale" resync branch as "reserved-name files pass, and nothing else is present" — but a genuinely fully-migrated directory always also has `MEMORY.md` (the index), which isn't itself a reserved topic filename and so never passes the scaffold check. Every real resync-eligible directory would therefore have incorrectly fallen into the ABORT branch instead. Found by live-testing the three-branch logic against constructed fixtures (a real 8-topic-file-plus-index directory, a genuinely ambiguous mixed directory, and a clean no-collision directory) before shipping, not by inspection — fixed by excluding `MEMORY.md` specifically from the "nothing else" condition, then re-verified all three branches resolve correctly.
- **`/mempenny:memory-shard-roll` and `/mempenny:memory-curate` both rewritten as mechanical Bash extraction (see "fixed differently from migration" above).** Two safety properties added beyond what the old design had, both validated with real fixtures (happy path, byte-for-byte independent cross-check, and the failure path):
  - **Shard-roll's structural guard:** the mechanical script assumes years appear in the file in non-increasing order (current year's months, then each closed year as one contiguous block, oldest last) — true for any file this codebase actually wrote, but the script explicitly checks it and fails closed (`SHARD-ROLL FAILED: structural check failed`) rather than silently mis-slicing a file that doesn't hold that shape, since a wrong assumption here would misfile content between shards rather than just fail loudly.
  - **Curate's two different failure treatments, refined after the first draft tolerated both the same way:** a locked entry the row-level re-check catches (independent of what the classification subagent decided) fails toward KEEP, tolerated up to 5% of rows — a legitimate, expected-to-occasionally-fire safety net. An entry *detected* in the file but missing from the classification table hard-fails the whole run immediately instead of being tolerated — the table is generated from this same file minutes earlier, so that mismatch is a structural red flag (most likely a heading-shaped line inside another entry's body fooling the boundary scan), not something to paper over by guessing. Neither of these covers an entry that's never *detected* in the first place — see "Known limitation" further below.
  - Neither script needed batching or multi-subagent coordination the way migration did — `sed -n` line-range extraction has no meaningful output-size ceiling, so the existing single-subagent-per-command architecture (Step 7 for shard-roll, Step 6 for curate) didn't need to change, only what that one subagent does internally.
- **2026-07-03, found via a third real-world migration ("game3", 75 files/728KB), not caught by the two prior rounds because neither had this project's own frontmatter shape *combined with* this dataset's very long single-line paragraphs and heavy `MEMORY.md`-index-bullet cross-referencing at once:** two real, distinct bugs, not one. **(1) Write-chunk scaling:** Phase B chunks could still hit real output-token-ceiling failures at the existing `{CHUNK_SIZE_CAP}`, even with the append-fragment mechanism (added in the v1.1.2 fix) working exactly as designed — the growing-file-reproduction problem that fix targeted wasn't the only path to the same ceiling; a chunk's own reasoning and cross-verification cost scales with how much it's individually relocating, not just its byte count. Fixed by giving Phase B its own, tighter `{WRITE_CHUNK_CAP}` (65,000 bytes AND at most 9 source files, whichever binds first) separate from Phase A's read-side `{CHUNK_SIZE_CAP}`. **(2) Conservation-check accuracy:** exact line matching had no exemption for old-file frontmatter (the `metadata:`-nested shape from the fix two entries above), so every old file's ~5 frontmatter lines registered as false `MISSING` — proven with a minimal, fully reproducible 2-file test where a byte-perfect body migration still failed with `TOTAL_MISSING=10`. Separately, exact matching couldn't verify a legitimate sub-line content split (a long paragraph divided mid-sentence across two topics — this dataset's paragraphs ran past 3,000 characters on a single raw line in places) or independently-routable `MEMORY.md` index bullets landing in different topics, without both registering as unaccounted for. On the real dataset this produced 735 false `MISSING` lines and an unnecessary (but correct, given the check as it then existed) rollback. Root-caused via a from-scratch redo using a word-coverage-based checker, which also surfaced a third, independent problem: Phase A's classify step can silently under-classify a file, with no signal distinguishable from a correct run — one 22.7KB file received rows for only a few short paragraphs and no `worklog.md` row at all, despite being overwhelmingly worklog-shaped content. "Every file appears in at least one classify row" is necessary but not sufficient; it can't detect a file that's only *partially* classified.
- **Same date, pre-deploy review round (code-review + pentest + Fable) on the first draft of the above two fixes found several more real issues before anything shipped — recorded honestly, matching the curate/shard-roll precedent above, since the first draft of a fix is not automatically safe just because it addresses the bug that motivated it:**
  1. **Execution-breaking, not just a hardening gap (Fable):** the first draft of the mechanical-append write script set `$FRAGMENT_PATH` in one Bash call and referenced it in a second, separate Bash call. Shell state does not persist between separate Bash tool calls in this harness — only the working directory does — so the second call would fail on an unbound variable, with neither of the write prompt's two defined success/failure sentinels ever printed. Fixed by having the first call print the resolved path explicitly and instructing the literal path string, not a variable reference, to be substituted into the second script.
  2. **A CRITICAL fail-open in the conservation check's own dispatch logic (Fable + Pentest, independently, Pentest with a working repro):** the three-pass awk script identified which of its three inputs (new-corpus lines, new-corpus words, old-corpus lines) it was reading by counting `FNR==1` transitions — a counter that never increments for an input that turns out to be completely empty. An empty new-corpus word list (Pentest's concrete trigger: the first draft's `clean()` function only recognized ASCII `[a-z0-9]` as word characters, so a topic written mostly in a non-Latin script stripped to nothing) misrouted every later input into the wrong pass, and the script printed `TOTAL_MISSING=0` — a false "nothing lost" — without ever actually running the comparison. Fixed two ways: dispatch now compares `FILENAME` against each input's own path directly (no counter, no empty-file blind spot), and `clean()` now trims a fixed set of known ASCII punctuation from each word's edges instead of allow-listing ASCII letters, so non-Latin-script content survives as real, comparable words instead of being zeroed out.
  3. **The coverage-fallback's leniency had no floor, letting real single-payload-token loss through (Fable + Pentest + Code-review, three independent repro shapes converging on the same root cause):** the first draft tolerated any line with ≤1 individually-uncovered word, or ≥90% coverage otherwise. Pentest confirmed a bare, ≥30-character single-token line (a URL, path, or identifier) could be entirely dropped and still pass, since its one uncovered word trivially satisfied "≤1." Fable found the same failure shape in "common prose plus exactly one distinctive payload token" — a very ordinary real line shape (any `- [text](file.md) — description` index bullet). Code-review's sharpest finding: a single word like "not" dropped from an otherwise-perfect, grammatically intact sentence can silently invert its meaning, and — because the word is common enough to coincidentally exist elsewhere in almost any real corpus — this specific failure shape survives even a corrected design; see "Known limitation" below. Fixed (for the first two shapes, not the third): coverage is now computed after excluding a small, fixed list of connective/structural words (articles, prepositions, common copulas, "see"/"related"/"note" — never negations, modals, or quantifiers, which are exactly the words most likely to carry logical weight if altered), and every remaining word must be found, full stop — no percentage or count tolerance. Validated this didn't just trade one false-positive class for another: re-running against the real 75-file dataset with the stricter rule initially surfaced 5 new false positives, traced to two further, narrower issues — a hard-wrapped old source line rejoined during relocation loses exactly the one word that sat at the old line break (fixed: tolerate exactly 1 uncovered word, but only once a line has ≥6 distinctive words, so losing one is never most of a short line's content), and markdown's `[text](url)` syntax glues the last word of the link text onto the url as one token since there's no space before `(` (fixed: insert the missing boundary before tokenizing). A decorative-only line (e.g. a `═══` divider) is now also exempted outright, having no alphanumeric content to lose. Re-validated against 10 controlled fixtures and the real dataset (both before and after the redo's own manual reconciliation) with zero false positives and zero false negatives once all of the above landed together.
  4. **A whole old file could be silently dropped by an unclosed frontmatter fence (Fable + Pentest + Code-review, independently confirmed):** the frontmatter-stripping function set its "inside frontmatter" flag on a line-1 `---` and only cleared it on a later `---` — a file whose leading `---` has no matching close (malformed, or simply not frontmatter at all) had its entire body silently withheld from the comparison on that side. Fixed: lines seen while the flag is set are now buffered rather than dropped, and flushed as real content if end-of-file arrives before a closing fence is ever found.
  5. **A symlink at the write target during an `APPEND` chunk had no guard (Pentest + Code-review, Pentest with working write-through and read-through repros):** a topic needing several sequential chunks leaves real wall-clock time between one chunk finishing and the next starting, long enough for something else to replace the target with a symlink in between. The first draft's newline-check and `cat` both dereferenced the target with no `-L` check anywhere, unlike this codebase's established discipline elsewhere (the lock-marker and `settings.json` guards). Fixed: an explicit symlink check on both the `CREATE` and `APPEND` branches, and the append landing no longer mutates the target directly at all (not even to add a trailing newline) — the newline and the fragment are composed together into the temp file in one step, closing the window entirely rather than narrowing it.
  6. **An unvalidated topic name reaching raw Bash string construction (Pentest, with a confirmed command-execution repro) — and the obvious first fix for it doesn't actually work:** `{TOPIC_FILENAME}` ultimately traces back to a classify subagent's own reading of untrusted file content, and reached double-quoted path construction with no allow-list check. The instinctive fix — a `case "{TOPIC_FILENAME}" in ...` guard inside the same script — was proven not to work: every placeholder in these scripts is substituted into the source text before Bash ever parses it, so an embedded `$(...)` fires the moment Bash evaluates the double-quoted string, before the case comparison itself ever runs, match or no match. (Confirmed directly: `case "x$(touch proof)" in ...; esac` runs the `touch` regardless of which arm matches.) Fixed at the only point it can actually be fixed — validated once, upstream, when Phase B forms topic groups from the classify output, the same place every other structurally-trusted placeholder in this file (starting with `{MEMORY_DIR}`) is already validated before ever reaching a script.
  7. **A failed migration's own documented recovery path didn't actually work (Fable + Code-review, independently):** Phase B's failure handling left any topic file that was mid-sequence (started but not yet complete) in place on disk, on a rationale — "APPEND mode needs it to exist if a retry resumes it" — that no longer matched reality once the previous fixes ruled out inline chunk retries entirely (the only supported recovery is a full re-run of the whole command). A full re-run's own collision pre-check would abort on exactly those leftover files, since they pass the topic-scaffold rubric while the old flat files are still also present. Fixed by tracking every topic that got at least one chunk confirmed written this run (not just the ones that reached full completion) and cleaning up all of them on any failure, so the re-run path the file itself tells the user to use is the one that actually works.
- **Verifying finding 3's fix, the same day, found a further real gap in it before shipping — a second-order finding from the review round verifying its own first round's fixes, not a new independent review pass:** the "tolerate exactly 1 uncovered word once a line has ≥6 distinctive words" exception had no way to tell a genuine hard-wrap reflow (the rest of the line demonstrably intact and adjacent) from a whole line that vanished outright while most of its individual words happened to coincidentally exist elsewhere in an unrelated sentence — both present identically as "exactly 1 uncovered word out of many." Confirmed with a realistic, non-contrived repro: an ordinary `- [text](file.md) — description` index bullet, entirely absent from the new corpus, where 9 of its 10 distinctive words nonetheless existed individually (scattered, non-contiguously) in unrelated new-file sentences — passed as `REORDERED`, non-fatal. Fixed by requiring actual positive evidence before granting that one-word tolerance, not just an absence of contrary evidence: a real 4-word run from immediately beside the uncovered word's position must be found intact and contiguous somewhere in the new corpus (an *exact* multi-word phrase recurring by chance is vanishingly unlikely, unlike any single common word). Implementation note: the new corpus, joined into one line for this contiguous-phrase search, is large enough on a real directory to exceed the shell's command-line argument size limit if passed to awk as a `-v` value directly (`Argument list too long`, hit empirically) — written to a temp file and read as a 4th awk input instead. Re-validated against 11 fixtures (10 prior + this one) and a live re-run of the real 75-file dataset with zero false positives and zero false negatives — and this specific re-run caught one genuine, previously-undetected small content gap in that already-migrated real directory (a single dropped list item inside a longer archive-manifest line, unrelated to any bug in `clean.md` itself — the file's own earlier conservation-check passes had all been lenient enough to miss it), fixed directly in that project's memory once confirmed.
- **Known limitation, accepted rather than chased further (parallel to curate/shard-roll's fence-parity limitation above):** even with phrase confirmation, the conservation check's word-coverage fallback cannot, even in principle, distinguish "a common word coincidentally exists elsewhere in the corpus because it's common" from "this specific occurrence of that word is the one that used to be in this sentence" — for the *zero*-uncovered-words path specifically, where phrase confirmation never triggers because there's no uncovered word to confirm around. A single connective, negation, or modal word altered within an otherwise fully-intact, information-dense sentence can still pass as `REORDERED` if that word happens to appear anywhere else in the new corpus — which, for sufficiently common words, it usually does. Closing this fully would require verifying local phrase-level continuity around *every* line, not just the one-word-tolerance path — a fundamentally different (and, at the deterministic-Bash-script scale this check is required to run at, far more expensive) technique. Mitigated, not solved: the fallback only ever engages after an exact-match attempt fails, so any change that also happens to alter surrounding punctuation, spacing, or word order is still caught; only a single-word substitution or deletion that leaves everything else byte-identical, with that one word coincidentally present elsewhere, can slip through. `TOTAL_REORDERED` is reported in every run specifically so an unusually high count is visible to whoever reads the report, same rationale as `EXCLUDED_AS_FENCED` above. The pre-existing full-directory backup, taken before any migration write and untouched regardless of outcome, remains the actual backstop for this specific residual risk — `/mempenny:restore` recovers from it same as anything else this check might miss.
